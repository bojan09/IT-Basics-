🐳 Docker

Package your app and everything it needs into a portable, self-contained unit that runs anywhere.

📖 EXPLAIN LIKE I'M A BEGINNER

What is Docker?

💡 THE ANALOGY Imagine you have a recipe (your app) and need to cook it in many different kitchens (computers). Without Docker, each kitchen might be missing different ingredients. Docker is like giving every kitchen a standardized, fully-stocked container — same ingredients, same tools, every time. Your app works the same everywhere.

Before Docker, developers would say "It works on my machine!" — a nightmare when code ran perfectly locally but crashed on the server. Docker solves this by packaging your app with all its dependencies into an isolated container.

⚖️ CONTAINERS vs VIRTUAL MACHINES

This is one of Docker's biggest advantages — containers are much lighter than VMs.

🐳
Docker Container

Shares the host OS kernel. Only packages the app and its dependencies.

  • Starts in milliseconds
  • Uses MB of disk space
  • Dozens can run on one machine
  • Portable across any OS
🖥️
Virtual Machine

Includes a full OS inside. Requires a hypervisor to run.

  • Takes minutes to start
  • Uses GB of disk space
  • Few can run on one machine
  • Stronger isolation
🧠 CORE CONCEPTS

Images, Containers, Registries

1
Dockerfile — the recipe
A text file with instructions for building your image. Like a recipe telling Docker exactly what ingredients to include.
2
Image — the blueprint
A read-only snapshot built from a Dockerfile. Like a template. You can share images via Docker Hub (a public registry).
3
Container — the running instance
A live, running copy of an image. You can run many containers from one image — like printing many copies from one master template.
4
Registry — the library
Where images are stored and shared. Docker Hub is the public one. Companies often run private registries (AWS ECR, GCR, etc.).
📄 A SIMPLE DOCKERFILE
DOCKERFILE
# Start from an official Python base image
FROM python:3.11-slim

# Set the working directory inside the container
WORKDIR /app

# Copy dependencies file and install them
COPY requirements.txt .
RUN pip install -r requirements.txt

# Copy the rest of your app code
COPY . .

# Tell Docker which port to expose
EXPOSE 8000

# The command to run when the container starts
CMD ["python", "app.py"]
⚡ ESSENTIAL COMMANDS
BASH
# Build an image from a Dockerfile
docker build -t my-app:latest .

# Run a container from an image
docker run -d -p 8000:8000 my-app:latest

# List running containers
docker ps

# View container logs
docker logs <container-id>

# Execute a command inside a running container
docker exec -it <container-id> bash

# Stop and remove a container
docker stop <container-id> && docker rm <container-id>

# Pull a public image from Docker Hub
docker pull nginx:latest
🔗 DOCKER COMPOSE

Running multiple containers together

Real apps need multiple services — a web server, a database, maybe a cache. Docker Compose lets you define all of them in one file.

🌐
Web Container
nginx:80
⚙️
App Container
python:8000
🗄️
DB Container
postgres:5432
⚡
Cache Container
redis:6379
DOCKER-COMPOSE.YML
version: '3.9'
services:
  web:
    image: nginx:latest
    ports: ["80:80"]
  app:
    build: .
    ports: ["8000:8000"]
    depends_on: [db, redis]
  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret
  redis:
    image: redis:7
BASH
# Start all services
docker compose up -d
# Stop all services
docker compose down
⚠️ COMMON MISTAKES

What beginners get wrong

❌ MISTAKE 1 Running everything as root. Containers should run as a non-root user. Add USER appuser in your Dockerfile.
❌ MISTAKE 2 Storing data inside containers. Containers are ephemeral — data inside them disappears when the container stops. Use volumes for persistent data.
❌ MISTAKE 3 Using the :latest tag in production. nginx:latest will grab whatever's newest — it might break your app. Pin specific versions: nginx:1.25.3.
❌ MISTAKE 4 One giant container for everything. The Docker philosophy is one process per container. Run your app, database, and cache in separate containers.
💡 Quick Summary — Docker
Docker packages your app + dependencies into portable containers that run identically anywhere.
A Dockerfile defines how to build an image. An image is a blueprint. A container is a running instance.
Containers share the host kernel — they're faster and lighter than VMs, but offer less isolation.
Use Docker Compose for multi-container apps (web + database + cache).
Always use specific version tags, non-root users, and volumes for data in production.
