☸️ Kubernetes

The container orchestration system that automatically manages, scales, and heals containerized applications.

📖 EXPLAIN LIKE I'M A BEGINNER

What is Kubernetes?

💡 THE ANALOGY Docker gives you containers. Kubernetes is the shipping company that manages fleets of those containers. It decides which ship (server) carries which container, automatically replaces broken containers, scales up when demand spikes, and balances the load — all without you manually intervening.

🧠 CORE CONCEPTS

Pods, Deployments, Services

📦
Pod
The smallest unit. One or more containers that share networking and storage. Think of it as a "wrapper" around your container(s).
🚀
Deployment
Declares how many copies of your Pod should run. Kubernetes ensures this count is always maintained — it restarts crashed pods automatically.
🌐
Service
A stable network endpoint that routes traffic to pods. Since pods have dynamic IPs, services provide a fixed address for other apps to connect to.
⚙️
ConfigMap
Stores configuration data (like env variables) separately from your app code. Inject at runtime without rebuilding the image.
🔒
Secret
Like ConfigMap but for sensitive data — passwords, API keys, TLS certs. Base64-encoded (not encrypted by default — add encryption at rest in production).
📈
HPA
Horizontal Pod Autoscaler. Automatically scales pod count up or down based on CPU/memory usage or custom metrics.
📄 A DEPLOYMENT YAML
YAML
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3 # run 3 copies
  selector:
    matchLabels:
      app: my-app
  template:
    spec:
      containers:
      - name: my-app
        image: my-app:1.2.0
        ports:
        - containerPort: 8000
        resources:
          limits:
            memory: "256Mi"
            cpu: "500m"
⚡ ESSENTIAL kubectl COMMANDS
BASH
kubectl get pods # list pods
kubectl get deployments
kubectl describe pod <name> # details + events
kubectl logs <pod-name> # view logs
kubectl apply -f deployment.yaml # deploy/update
kubectl delete -f deployment.yaml # remove
kubectl scale deployment my-app --replicas=5
kubectl exec -it <pod> -- bash # shell into pod
⚠️ COMMON MISTAKES

What beginners get wrong

❌ MISTAKE 1Not setting resource limits. Without limits, one pod can consume all memory and crash the entire node. Always set resources.limits.
❌ MISTAKE 2Using :latest image tags. Same as with Docker — pin to a specific version. Rolling updates become unpredictable otherwise.
❌ MISTAKE 3Only running 1 replica in production. A single pod = a single point of failure. Run at least 2–3 replicas and spread across nodes.
❌ MISTAKE 4Storing secrets in ConfigMaps. ConfigMaps are plain text. Sensitive data belongs in Secrets — or better, a secrets manager (Vault, AWS Secrets Manager).
💡 Quick Summary — Kubernetes
Kubernetes orchestrates containers at scale — scheduling, healing, scaling automatically.
Pod = smallest unit (1+ containers). Deployment = manages pod replicas. Service = stable network endpoint.
The Control Plane (API server, scheduler, etcd) makes decisions. Nodes run the actual workloads.
Everything is declared in YAML and applied with kubectl apply -f.
Always set resource limits, use specific image tags, and run multiple replicas in production.
